# WSJT-Z Windows Build: Known Pitfalls (MSYS2/MinGW64)

Root-caused during the v3.0.0 first-release build cycle. Every error below
was hit in sequence and required an iterative CI fix. This document exists so
the next release does not repeat that cycle.

---

## Environment

| Item | Value |
|------|-------|
| Runner | `windows-latest` (GitHub Actions) |
| Build system | MSYS2 MinGW64 |
| Compiler | GCC/GFortran **14.2.0-3** (pinned — see Pitfall 5) |
| Build tool | Ninja |
| Qt | `mingw-w64-x86_64-qt5-*` via pacman |
| Packager | NSIS via CPack |

---

## Pitfall 1 — `mingw-w64-x86_64-hamlib` does not exist in MSYS2

**Symptom:** `pacman` exits with `error: target not found: mingw-w64-x86_64-hamlib`.

**Root cause:** Hamlib is not packaged in the MSYS2/MinGW64 repository. It must be built from source.

**Fix:** Build Hamlib 4.5.5 from the upstream tarball inside the MSYS2 shell:

```bash
curl -L https://github.com/Hamlib/Hamlib/releases/download/4.5.5/hamlib-4.5.5.tar.gz | tar xz
cd hamlib-4.5.5
./configure --prefix=/mingw64 --disable-static --without-cxx-binding --disable-winradio
make -j$(nproc) && make install
```

**Required pacman packages:** `autoconf automake libtool make`

## Pitfall 2 — readline API mismatch during Hamlib build

**Symptom:** Hamlib `rigctl.c` fails with:

```
error: 'rl_readline_name' undeclared
error: too many arguments to function 'readline'
error: too many arguments to function 'read_history'
```

**Root cause:** MSYS2 ships `libedit` which provides a partial readline-compatible stub. The stubs have different function signatures from GNU readline. Hamlib's `configure` detects `libedit` as "readline support" and enables it, then fails to compile against the wrong API.

**Fix:** Install `mingw-w64-x86_64-readline` (GNU readline) **before** configuring Hamlib. GNU readline has the correct signatures and takes precedence over libedit.

**Note:** `--disable-readline` is **not** a recognised option in Hamlib 4.5.5 `configure` — it is silently ignored with a warning.

## Pitfall 3 — OmniRig: CMake fatal error at configure time

**Symptom:**
```
CMake Error at CMakeLists.txt:924 (message):
  You need to install OmniRig on this computer
```

**Root cause:** The `WSJT_WITH_OMNIRIG` CMake option defaults to `ON`. OmniRig is a Windows COM/ActiveX server that must be installed on the build machine. CI runners have no COM servers registered.

**Fix:** Pass `-DWSJT_WITH_OMNIRIG=OFF` to CMake configure.

---

## Pitfall 4 — OmniRig: C++ compile errors after CMake flag added

**Symptom** (after Pitfall 3 fix):
```
fatal error: OmniRig.h: No such file or directory
error: 'OmniRigOneId' was not declared in this scope
error: 'OmniRigTransceiver' does not name a type
```

**Root cause:** `OmniRig.h` is generated by `qaxcontainer` only when `WSJT_WITH_OMNIRIG=ON`. `TransceiverFactory.cpp` had three blocks guarded by `#if defined(WIN32)` only — not checking whether OmniRig was disabled at the CMake level.

**Fix:** Three-part change:

1. `CMakeLists.txt`: add compile definition `WSJT_NO_OMNIRIG` to the `wsjt_qt` target when `WIN32 AND NOT WSJT_WITH_OMNIRIG`.
2. `Transceiver/TransceiverFactory.cpp`: change all three `#if defined(WIN32)` guards to `#if defined(WIN32) && !defined(WSJT_NO_OMNIRIG)`:
   - The `#include "OmniRigTransceiver.hpp"` line
   - The `OmniRigTransceiver::register_transceivers(...)` call
   - The `case OmniRigOneId:` / `case OmniRigTwoId:` switch blocks and their enum values

All three locations must be updated together. Fixing only one or two causes a follow-on error from the remaining unguarded block.

## Pitfall 5 — GCC 15 `.tls_common` regression in Fortran code

**Symptom:** Assembler error on Fortran TLS variables:
```
Error: .tls_common directive is not supported
```

**Root cause:** GCC 15 changed the default thread-local storage model for MinGW targets. Fortran modules with `SAVE` variables compiled with OpenMP emit `.tls_common` directives that GCC 15's MinGW assembler rejects.

**Fix:** Downgrade **only the compiler binaries** to 14.2.0-3. Keep `gcc-libs` at the current version (runtime ABI). The `--nodeps` flag prevents pacman from pulling in the older runtime libs:

```bash
BASE=https://repo.msys2.org/mingw/mingw64
pacman -U --noconfirm --nodeps \
  "${BASE}/mingw-w64-x86_64-gcc-14.2.0-3-any.pkg.tar.zst" \
  "${BASE}/mingw-w64-x86_64-gcc-fortran-14.2.0-3-any.pkg.tar.zst"
```

**Why 14.2.0-3 specifically?** It is the newest version that predates the TLS regression, and new enough to support `-fallow-argument-mismatch` (GFortran 10+, required for Pitfall 7). Do not go lower than GFortran 10.

**This step must run before building Hamlib**, as Hamlib is also compiled with the same GCC.

## Pitfall 5b — Runtime DLL mismatch: `__emutls_v_ZSt11__once_call` entry point not found

**Symptom:** After installing, `wsjtx.exe` immediately crashes with:
```
wsjtx.exe — Entry Point Not Found
The procedure entry point __emutls_v_ZSt11__once_call could not be located
in the dynamic link library C:\WSJT\wsjtx\bin\wsjtx.exe
```

**Root cause:** The original Pitfall 5 fix only downgraded the compiler
binaries (`gcc`, `gcc-fortran`) to 14.2.0-3, but left `gcc-libs` at
the current GCC 15.x version. `gcc-libs` provides the runtime DLLs
(`libstdc++-6.dll`, `libgcc_s_seh-1.dll`, `libgfortran-5.dll`,
`libgomp-1.dll`) that are bundled into the NSIS installer.

The result: `wsjtx.exe` was compiled with GCC 14 and references the GCC 14
ABI symbol `__emutls_v_ZSt11__once_call`, but the installer shipped the GCC
15 `libstdc++-6.dll` which no longer exports that symbol under the same name.

**Fix:** Do NOT attempt `pacman -U gcc-libs` — modern MSYS2 renamed the provides to `cc-libs` and every installed package depends on it, causing a chain of conflicts. Instead, **overwrite the DLL files in `/mingw64/bin/` directly via tar**, bypassing the pacman database:

```bash
BASE=https://repo.msys2.org/mingw/mingw64
curl -sL "${BASE}/mingw-w64-x86_64-gcc-libs-14.2.0-3-any.pkg.tar.zst" \
  | zstd -dc | tar --overwrite -x -C / \
      mingw64/bin/libstdc++-6.dll \
      mingw64/bin/libgcc_s_seh-1.dll \
      mingw64/bin/libgomp-1.dll \
      mingw64/bin/libquadmath-0.dll \
      mingw64/bin/libatomic-1.dll
```

**WARNING — do NOT use `tar --overwrite` to `/mingw64/bin/`.**
This breaks cmake, ninja, and every other tool compiled against the current GCC, because they immediately fail to load with exit code 127 (their `libstdc++-6.dll` no longer exports the GCC 16 symbols they need).

**Final working approach — isolated DLL directory:**

1. Extract the GCC 14 DLLs to a separate directory (`/tmp/gcc14dlls/`):
```bash
BASE=https://repo.msys2.org/mingw/mingw64
mkdir -p /tmp/gcc14dlls
curl -sL "${BASE}/mingw-w64-x86_64-gcc-libs-14.2.0-3-any.pkg.tar.zst" \
  | zstd -dc | tar -x --strip-components=2 -C /tmp/gcc14dlls \
      mingw64/bin/libstdc++-6.dll \
      mingw64/bin/libgcc_s_seh-1.dll \
      mingw64/bin/libgomp-1.dll \
      mingw64/bin/libquadmath-0.dll \
      mingw64/bin/libatomic-1.dll
```

2. Pass the directory to CMake: `-DGCC_RUNTIME_OVERRIDE_DIR=/tmp/gcc14dlls`

3. `bundle_fixup/CMakeLists.txt` prepends `GCC_RUNTIME_OVERRIDE_DIR` to `fixup_library_dirs` before the Qt/system directory, so `fixup_bundle` finds and bundles the GCC 14 DLLs first. Build tools still use GCC 16 from `/mingw64/bin/` — unaffected.

**Rule:**
- `gcc` + `gcc-fortran`: `pacman -U --nodeps` (safe — no reverse deps)
- `gcc-libs` DLLs: **do not attempt to inject** — see Pitfall 5c

## Pitfall 5c — Second DLL ABI crash: `_ZSt15__get_once_callv` not found in `libicuuc78.dll`

**Symptom:** At the end of the NSIS installer, or on first launch:
```
wsjtx.exe — Entry Point Not Found
The procedure entry point _ZSt15__get_once_callv could not be located
in the dynamic link library C:\WSJT\wsjtx\bin\libicuuc78.dll
```

**Root cause:** This is the inverse of Pitfall 5b. `libicuuc78.dll` is Qt's ICU
library, compiled by MSYS2 with GCC 16. GCC 16's `libicuuc78.dll` imports
`_ZSt15__get_once_callv` (`std::__get_once_call()`) from `libstdc++-6.dll`.
If we replaced `libstdc++-6.dll` with GCC 14's version (Pitfall 5b fix),
that GCC 14 DLL does not export `_ZSt15__get_once_callv` → ICU crashes.

**The catch-22:** Using GCC 14 libstdc++ satisfies wsjtx.exe but breaks ICU.
Using GCC 16 libstdc++ satisfies ICU but breaks wsjtx.exe. There is no single
shared `libstdc++-6.dll` that satisfies both.

**Definitive fix: statically link the GCC runtime into the executables.**

Add to `CMakeLists.txt`:
```cmake
if (WIN32 AND MINGW)
  set (CMAKE_EXE_LINKER_FLAGS
       "${CMAKE_EXE_LINKER_FLAGS} -static-libgcc -static-libstdc++ -static-libgfortran")
endif ()
```

This bakes GCC 14's `libstdc++`, `libgcc`, and `libgfortran` directly into each
`.exe`. The executables no longer import from `libstdc++-6.dll` at all. The MSYS2
packages (Qt, ICU, Boost) continue to use GCC 16's shared `libstdc++-6.dll`
without conflict — each side has its own copy of the runtime.

**Consequence:** Each `.exe` is ~3–5 MB larger. No `libstdc++-6.dll`,
`libgcc_s_seh-1.dll`, or `libgfortran-5.dll` need to be bundled in the
installer at all.

## Pitfall 6 — `NFFT` undeclared in `map65/libm65/decode0.f90`

**Symptom:**
```
Error: Variable 'nfft' cannot appear in the expression at (1)
  real*4 dd(4,NSMAX),ss(4,322,NFFT),savg(4,NFFT)
```

**Root cause:** `decode0.f90` uses `NFFT` as an explicit array-shape bound without declaring it. Older GCC accepted it as an implicit external reference from the calling scope. GFortran 14 enforces Fortran 90 standard: all names used in array bounds must be declared in the same scope.

**Fix:** Add `parameter (NFFT=32768)` to `decode0.f90`, matching the value in `m65.f90`'s `common/datcom/` definition.

This is also a latent correctness bug in the upstream — the two declarations were silently coupled through the compiler extension.

## Pitfall 7 — MAP65 Fortran type-punning rejected by GFortran 14

**Symptom:**
```
Error: Type mismatch in argument 'dd' at (1); passed INTEGER(1) to REAL(4)
Error: Rank mismatch in argument 'm65com' at (1) (rank-1 and scalar)
```
in `map65/libm65/m65a.f90`.

**Root cause:** MAP65 uses a shared byte buffer (`m65com`, `INTEGER(1)`) that is passed to subroutines expecting `REAL(4)` and `INTEGER(4)` arrays. This is a pre-Fortran-90 memory-aliasing pattern (equivalent to C's union via pointer cast). GFortran 14 treats argument type mismatches as hard errors by default.

**Fix:** Add `-fallow-argument-mismatch` to `General_FFLAGS` in `CMakeLists.txt`:

```cmake
set (General_FFLAGS "-Wall -Wno-conversion -fno-second-underscore -fallow-argument-mismatch")
```

This flag (introduced in GFortran 10) demotes type-mismatch errors to warnings for legacy code. Do not attempt to rewrite the MAP65 shared-memory scheme — it spans dozens of subroutines and the risk of regression outweighs the benefit.

## Pitfall 8 — `fftwf_*_threads` undefined at link time

**Symptom:**
```
undefined reference to `fftwf_init_threads'
undefined reference to `fftwf_plan_with_nthreads'
undefined reference to `fftwf_cleanup_threads'
```
when linking `jt9.exe`.

**Root cause:** `CMake/Modules/FindFFTW3.cmake` contained:

```cmake
# If using threads, we need to link against threaded libraries as well - except on Windows.
if (NOT WIN32 AND _use_threads)
```

The `NOT WIN32` condition was added assuming the old JTSDK environment where the monolithic `fftw3f.dll` already included thread support. MSYS2's `mingw-w64-x86_64-fftw` package is modular — `libfftw3f_threads.dll.a` is a separate file not linked by default.

**Fix:** Remove `NOT WIN32` from the condition:

```cmake
if (_use_threads)
```

The threads library is now linked on all platforms when the `threads` component is requested, which is correct behaviour everywhere.

## Pitfall 9 — OmniRig CAT control: re-enabling correctly (not just disabling it)

`WSJT_WITH_OMNIRIG=OFF` is the safe fallback (Pitfalls 3/4) but loses
OmniRig as a selectable rig in the CAT dropdown. To enable it correctly:

**Three separate problems, not one:**

1. **`Qt5AxContainer` package name.** It DOES exist for MinGW64 in MSYS2 —
   `mingw-w64-x86_64-qt5-activeqt`. A naive `grep` for "activeqt" against
   the raw repo listing can miss it if the match pattern is wrong; verify
   directly against `packages.msys2.org` or by downloading the `.zst` and
   listing its contents.

2. **`dumpcpp` binary name.** The package installs `dumpcpp-qt5.exe`, not
   `dumpcpp.exe`. Both `CMakeLists.txt` (`find_program (DUMPCPP dumpcpp)`)
   and `CMake/Modules/QtAxMacros.cmake`
   (`find_program (DUMPCPP_Executable dumpcpp.exe)`) hardcode the plain
   name. Fix: `cp /mingw64/bin/dumpcpp-qt5.exe /mingw64/bin/dumpcpp.exe`
   in the CI workflow before configuring.

3. **CLSID registry lookup and the WOW64 split.** The original code does
   `dumpcpp -getfile {4FE359C5-A58F-459D-BE95-CA559FB4F270}`, which
   requires OmniRig installed AND registered on the build machine.
   OmniRig is a 32-bit Delphi COM server. Our toolchain is 64-bit MinGW.
   A 64-bit process querying `HKEY_CLASSES_ROOT\CLSID\{...}` reads the
   *native 64-bit registry view*, not the `WOW6432Node` view where a
   32-bit COM server's self-registration normally lands. Installing
   OmniRig on the runner does not guarantee the lookup succeeds.

   **Fix: bypass the registry lookup entirely.** OmniRig's type library
   is checked directly into its GitHub source repo
   (`VE3NEA/OmniRig/OmniRig.tlb`). `dumpcpp` accepts a direct file path
   as an alternative to `-getfile {CLSID}` — both produce the same
   wrapper output. Added an `OMNIRIG_TLB_PATH` CMake override:

   ```cmake
   if (OMNIRIG_TLB_PATH)
     file (TO_CMAKE_PATH ${OMNIRIG_TLB_PATH} AXSERVERSRCS)
   else ()
     # original registry-based lookup, unchanged, for local dev builds
   endif ()
   ```

   CI step:
   ```bash
   curl -sL -o OmniRig.tlb \
     https://raw.githubusercontent.com/VE3NEA/OmniRig/master/OmniRig.tlb
   ```
   ```bash
   cmake ... -DOMNIRIG_TLB_PATH="$(cd .. && pwd)/OmniRig.tlb" ...
   ```

   This also avoids running a third-party installer in CI at all —
   no virus-scan-flagged `.exe`, no install state to manage.

4. **Narrowing conversion in `OmniRigTransceiver.cpp`.** Once the build
   gets far enough to compile this file, GCC 14 may reject:
   ```cpp
   switch (rig_->GetRxFrequency () - test_frequency)
     {
     case -5: ...
   ```
   with `narrowing conversion of '-5' from 'int' to 'long long unsigned int'`.
   This happens because `dumpcpp`'s generated wrapper for
   `GetRxFrequency()` is typed as returning an unsigned 64-bit value (the
   exact type depends on how dumpcpp interprets the IDL property type in
   the `.tlb`, which can differ from what the original WSJT-X authors saw
   on their MSVC + official-Qt-builds toolchain). Fix: cast the switch
   expression to a signed type, which is always safe for radio
   frequencies (comfortably within `qint64` range) and changes no
   behaviour:
   ```cpp
   switch (static_cast<qint64> (rig_->GetRxFrequency () - test_frequency))
   ```

**End-user requirement:** the compiled application can now offer OmniRig
as a CAT control option, but end users who want to actually *use* it must
separately install OmniRig themselves (`http://www.dxatlas.com/OmniRig/`)
— same as for any other WSJT-X/Z-family fork. We do not bundle OmniRig in
the installer; only its build-time type library is needed to compile the
wrapper code.

## Quick-reference checklist for future forks

Copy this into the PR description for any new WSJT-X/Z Windows CI setup:

```
Windows MSYS2 build checklist:
[ ] Hamlib NOT in pacman — build 4.5.5 from tarball with autotools
[ ] mingw-w64-x86_64-readline installed BEFORE Hamlib configure
[ ] gcc-libs + gcc + gcc-fortran ALL pinned to 14.2.0-3 via pacman -U --nodeps
    (pinning only gcc/gcc-fortran causes runtime DLL mismatch on install)
[ ] -DWSJT_WITH_OMNIRIG=OFF passed to CMake
[ ] WSJT_NO_OMNIRIG guards all 3 OmniRig blocks in TransceiverFactory.cpp
[ ] parameter (NFFT=32768) declared in map65/libm65/decode0.f90
[ ] -fallow-argument-mismatch in General_FFLAGS (CMakeLists.txt)
[ ] FindFFTW3.cmake: NOT WIN32 condition removed from threads block
[ ] libhamlib-utils in Linux dep install (rigctl needed by CPack)
```

## Why not JTSDK?

The official WSJT-X releases use JTSDK (WSJT Software Development Kit), a custom interactive installer providing a pinned build environment. JTSDK cannot be used in CI:

- No silent install mode compatible with GitHub Actions runners
- Assumes interactive Windows desktop with specific `C:\JTSDK64-Tools` paths
- Not available as a container image
- Tightly coupled to a specific Qt and Hamlib version with no upgrade path

MSYS2 provides equivalent tooling via `pacman` automation. The eight pitfalls above are the one-time mapping cost from JTSDK's pre-built environment to MSYS2's modular packages. All fixes are now baked into the source tree — future builds from this fork start clean.
